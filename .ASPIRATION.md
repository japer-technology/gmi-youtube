# Aspiration

## Why This Exists

YouTube is extraordinary at hosting content and inconsistent at delivering calm, intentional viewing.

My subscriptions, playlists, likes, and watch patterns represent years of human curation. That curation is the real asset. This repository exists to preserve it, structure it, and present it through an experience I control.

The goal is straightforward:

- Keep YouTube as the content source.
- Use GitHub as the processing and memory layer.
- Use GitHub Pages as the interface.
- Use an AI agent as the conversational control surface.

This is not a YouTube replacement. It is a personal station built on top of YouTube, operated through data, automation, and dialogue.

## Core Product Thesis

This repository should become a viable, durable solution for high-quality YouTube utilisation: a **personal TV station** that can be operated by simply talking to a personal YouTube AI Agent.

That means the system must do four things reliably:

1. **Collect** relevant YouTube signals (subscriptions, videos, playlists, live state, transcript availability, activity, comments).
2. **Normalize** those signals into stable, versioned resources in the repository.
3. **Curate** those resources into meaningful views, schedules, and recommendations.
4. **Present** the result as a calm, fast, static interface that works across screens.

If this works, opening the station should feel like turning on a channel designed for one person, not entering an engagement funnel.

## Product Definition

### What We Are Building

A personal media operating layer composed of:

- **YouTube Data API + OAuth** for source data
- **GitHub Actions** for scheduled ingestion, transformation, and publishing
- **Repository resources** (JSON/Markdown) as the source of truth for derived state
- **GitHub Pages** as the static front-end
- **Personal YouTube AI Agent** as the natural-language interface for control and curation

### What We Are Not Building

- A video hosting platform
- A social network
- A heavy backend with opaque runtime state
- A recommendation engine optimized for watch-time extraction

## Architectural Model

### 1) YouTube: Source Layer

YouTube remains the canonical source for:

- Channels and subscriptions
- Videos and metadata
- Playlists and ordering
- Live and premiere state
- Captions and transcript availability
- Activity signals
- Comment threads
- Search and discovery

### 2) GitHub: Processing + Memory Layer

GitHub Actions workflows perform scheduled and on-demand operations to:

- Retrieve source data
- Transform responses into stable, typed resources
- Track change over time
- Preserve historical snapshots
- Publish resources for frontend consumption

Git is the long-term memory.

### 3) GitHub Pages: Experience Layer

GitHub Pages provides a static, legible interface that:

- Renders guide, wall, channel, and discovery views
- Supports embed and fallback links
- Surfaces live/upcoming context
- Exposes transcript-aware search
- Supports multi-screen roles

### 4) AI Agent: Control Layer

The Personal YouTube AI Agent is the primary command surface.

The user should be able to say:

- “Show me what is live now from my science channels.”
- “Build a 3×3 channel wall for calm long-form viewing.”
- “Find a recent deep interview on AI safety.”
- “Update my evening lineup and prioritize these creators.”
- “What did I watch this week, and what did I skip?”

The agent should respond with concrete actions, grounded in repository data and YouTube API results.

## Boundary Clarity

| Boundary | Role | Responsibility |
|---|---|---|
| **YouTube** | Content source | Hosts videos/channels/playlists and exposes API resources |
| **GitHub Actions** | Processing engine | Fetch, transform, schedule, and publish data artifacts |
| **Repository** | Memory + governance | Versioned history, schemas, rules, curated state |
| **GitHub Pages** | Viewer interface | Static rendering of guides, walls, channels, search, playback entry points |
| **AI Agent** | Conversational operator | Interprets intent, orchestrates curation actions, answers with context |

## Experience Principles

The station experience should be:

- **Calm** — low-noise presentation, no compulsion mechanics
- **Legible** — clear structure, clear provenance, clear intent
- **Personal** — explicitly shaped by my subscriptions and preferences
- **Transparent** — inspectable pipeline and versioned changes
- **Portable** — reusable resource layer independent of any one UI
- **Composable** — one dataset powering many views and automations

## Product Surfaces

### TV Guide

A time-based programme guide that answers:

- What is on now?
- What is next?
- What aired recently?

It should support:

- Classic channel × time grid
- Genre/topic groupings
- Editorial blocks
- Live and premiere highlighting
- Local time-zone rendering

### Channel Wall

A configurable N×M channel matrix (1×1 through 10×10) for rapid situational awareness.

Requirements:

- Responsive layout across devices
- Saved layouts in repo-managed configuration
- Fast refresh from scheduled data updates
- Clear live/upcoming indicators

### Multi-Screen

Screens should coordinate without server complexity.

Possible roles:

- Guide screen
- Playback screen
- Wall screen
- Curator chat screen

Coordination can be lightweight (`BroadcastChannel`, URL state, localStorage). Multi-screen must enhance the experience, never become a requirement.

### Intelligent Curator

The AI Agent should evolve into an intelligent curator that can:

- Classify by topic, mood, format, and quality
- Read transcripts and summarize meaning
- Incorporate comment sentiment signals
- Detect channel behavior shifts
- Propose schedules and editorial blocks
- Manage lineups through conversation

### Viewing Receipt

The system should produce periodic viewing receipts (daily/weekly/on-demand):

- What was watched
- What arrived new
- What was skipped
- What the curator flagged
- How viewing patterns changed

Receipts should be calm, readable, configurable, and archived in-repo for long-term reflection.

## YouTube API Surface (Strategic)

The station should be built with full awareness of high-value API resources:

- `subscriptions`
- `channels`
- `playlistItems`
- `playlists`
- `videos` (details, statistics, liveStreamingDetails, player, topicDetails)
- `activities`
- `search`
- `commentThreads`
- `comments`
- `captions` (list/download)
- `videoCategories`
- `channelSections`
- `i18nLanguages`
- `i18nRegions`

### Quota Discipline

Daily quota is finite (default 10,000 units). The platform must treat quota as an editorial budget:

- Prioritize subscription intelligence first
- Spend expensive search calls intentionally
- Use transcript and caption calls selectively
- Preserve quota headroom for live checks and high-value user requests

## Data Model Ambition

The repository should become a high-integrity personal media dataset containing:

- Subscription graph and channel metadata
- Video records and temporal state
- Playlist structure and ordering
- Activity timelines
- Live/premiere schedules
- Transcript corpora and search indexes
- Comment samples and community context
- Curation tags and AI enrichments
- Viewing logs and receipts
- Historical guide snapshots

The interface can evolve. The dataset is the durable asset.

## Editorial Standard

Every feature should move the station toward:

- **Intentional curation over maximal ingestion**
- **Human readability over hidden complexity**
- **Reliable automation over brittle cleverness**
- **Product clarity over feature sprawl**

## Quality Bar for Future Work

Before implementing any feature, validate:

1. Does this make the station more personal?
2. Does this improve ownership and clarity?
3. Does this strengthen the data layer?
4. Does this preserve architectural boundaries?
5. Does this improve calmness, legibility, or usefulness?
6. Does this work in multi-screen contexts?
7. Does this respect quota and cron budget constraints?
8. Does this improve the AI curator’s effectiveness?
9. Does this improve receipt quality and reflective value?
10. Does this use the right YouTube API resource for the job?
11. Does this make the TV Guide and Channel Wall materially better?

If the answer is no, it is likely out of scope.

## Delivery Standard

To be considered viable, this repository should be able to demonstrate:

- Reliable scheduled ingestion
- Stable resource schemas
- Fast static rendering
- Useful guide and wall views
- Transcript-aware discovery
- Practical multi-screen coordination
- Actionable conversational control through the Personal YouTube AI Agent
- Clear viewing receipts and historical accountability

## Final Aspiration

I want this system to make YouTube utilisation excellent through ownership, not addiction.

I want my subscriptions to function like channels.
I want my playlists to function like programme blocks.
I want my station to be programmable by conversation.
I want my data, history, and decisions to remain mine.

**The aspiration is a personal TV station where YouTube provides the content, GitHub provides the infrastructure, and a personal YouTube AI Agent provides the interface for thought, control, and curation.**
