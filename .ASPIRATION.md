# Aspiration

## Why This Exists

YouTube is one of the best content libraries ever assembled and one of the worst environments for deliberate viewing.

The platform is exceptional at hosting video, but inconsistent at helping a person turn years of subscriptions, playlists, likes, and watch history into a calm, coherent, high-quality experience. That accumulated curation is the real asset. This repository exists to protect it, structure it, and make it genuinely useful.

The objective is simple:

- Keep YouTube as the content source.
- Use GitHub as the processing, memory, and governance layer.
- Use GitHub Pages as the delivery surface.
- Use a personal YouTube AI Agent as the conversational interface.

This is not an attempt to replace YouTube. It is an attempt to make YouTube finally work as a personal, intelligent, user-owned media system.

## Strategic Objective

This repository should become a credible, durable solution for excellent YouTube utilisation: a **personal TV station and viewing environment** that can be operated by simply talking to a personal YouTube AI Agent.

For that to be true, the system must do four things reliably:

1. **Collect** the right YouTube signals: subscriptions, channels, videos, playlists, live state, transcript availability, activity, and relevant conversation.
2. **Normalize** those signals into stable, versioned repository resources that can be trusted over time.
3. **Curate** those resources into meaningful schedules, interfaces, recommendations, and reflective summaries.
4. **Present** the result through a calm, fast, static experience that works well across devices and screens.

If this vision is executed properly, opening the station should feel less like entering an engagement funnel and more like turning on a channel that was designed for one person, by that person, with machine assistance.

## Product Definition

### What We Are Building

A personal media operating layer composed of:

- **YouTube Data API + OAuth** for source data and authenticated user context
- **GitHub Actions** for scheduled ingestion, transformation, enrichment, and publishing
- **Repository resources** (JSON and Markdown) as the source of truth for derived state
- **GitHub Pages** as the static presentation layer
- **Personal YouTube AI Agent** as the natural-language control surface for search, curation, and orchestration

### What We Are Not Building

- A video hosting platform
- A social network
- A heavy backend with opaque runtime state
- A recommendation engine optimised for watch-time extraction
- A product that hides its logic, trade-offs, or history from the user

## Product Promise

The promise of this repository is straightforward:

**Take the value already created by years of YouTube use and turn it into an owned, inspectable, programmable viewing system.**

That promise only holds if the system is:

- useful in daily practice, not only interesting in theory
- architecturally modest enough to maintain
- transparent enough to trust
- good enough in experience quality that a person would actually prefer it to default YouTube browsing for many use cases

## Architectural Model

### 1) YouTube: Source Layer

YouTube remains the canonical source for:

- Channels and subscriptions
- Videos and metadata
- Playlists and ordering
- Live and premiere state
- Captions and transcript availability
- Activity signals
- Comment threads
- Search and discovery

### 2) GitHub: Processing and Memory Layer

GitHub Actions should perform scheduled and on-demand operations to:

- retrieve source data
- transform API responses into stable typed resources
- track changes over time
- preserve historical snapshots
- publish resources for frontend consumption
- record the system's own operational history

Git is the long-term memory. The repository is not just where the code lives; it is where the media model, curation decisions, and historical record should live as well.

### 3) GitHub Pages: Experience Layer

GitHub Pages should provide a static, legible interface that:

- renders guide, wall, channel, and discovery views
- supports embedded playback and sensible fallback links
- surfaces live and upcoming context clearly
- enables transcript-aware search and retrieval
- supports multi-screen use without demanding multi-screen complexity

### 4) AI Agent: Control Layer

The personal YouTube AI Agent should be the primary command surface.

The user should be able to say:

- “Show me what is live now from my science channels.”
- “Build a 3×3 channel wall for calm long-form viewing.”
- “Find a recent deep interview on AI safety.”
- “Update my evening lineup and prioritise these creators.”
- “What did I watch this week, and what did I ignore?”

The agent should respond with concrete actions and grounded answers derived from repository state and current YouTube data, not vague suggestions or simulated understanding.

## Boundary Clarity

| Boundary | Role | Responsibility |
|---|---|---|
| **YouTube** | Content source | Hosts videos, channels, playlists, live signals, comments, captions, and search surfaces |
| **GitHub Actions** | Processing engine | Fetches, transforms, schedules, enriches, and publishes data artifacts |
| **Repository** | Memory and governance | Stores schemas, versioned history, curated state, rules, receipts, and product intent |
| **GitHub Pages** | Viewer interface | Renders guides, walls, channels, search, summaries, and playback entry points |
| **AI Agent** | Conversational operator | Interprets intent, orchestrates curation, answers questions, and proposes actions with context |

Architectural discipline matters. Each layer should stay legible. If a feature blurs boundaries, increases hidden state, or introduces runtime opacity without a compelling payoff, it is probably the wrong move.

## Experience Principles

The station experience should be:

- **Calm** — low-noise, low-pressure, free from compulsion mechanics
- **Legible** — clear in structure, provenance, and purpose
- **Personal** — explicitly shaped by my subscriptions, interests, and editorial choices
- **Transparent** — inspectable in pipeline behaviour, outputs, and historical change
- **Portable** — useful as a resource layer independent of any one interface
- **Composable** — able to power multiple views, automations, and workflows from one dataset
- **Trustworthy** — reliable enough that the user can build habits on top of it

## Product Surfaces

### TV Guide

A time-based programme guide that answers:

- What is on now?
- What is next?
- What aired recently?
- What is worth making time for?

It should support:

- classic channel-by-time grid layouts
- genre and topic groupings
- editorial blocks and named lineups
- live and premiere highlighting
- local time-zone rendering
- quick movement from overview to playback

### Channel Wall

A configurable N×M channel matrix (from 1×1 through 10×10) for rapid situational awareness.

Requirements:

- responsive layout across devices
- repo-managed saved layouts
- fast refresh from scheduled data updates
- clear live, upcoming, and stale-state indicators
- graceful behaviour when embeds are unavailable

### Multi-Screen

Multi-screen support should improve the experience without turning the product into a distributed-systems project.

Possible roles:

- guide screen
- playback screen
- wall screen
- curator chat screen

Coordination can remain lightweight (`BroadcastChannel`, URL state, `localStorage`). Multi-screen is an enhancement, not a prerequisite.

### Intelligent Curator

The AI Agent should evolve into an intelligent curator that can:

- classify content by topic, mood, format, and quality
- read transcripts and summarise meaning
- incorporate comment and community signals where useful
- detect meaningful channel behaviour shifts
- propose schedules, editorial blocks, and viewing paths
- manage lineups through natural conversation

Its job is not to mimic a generic recommendation feed. Its job is to become a competent personal editor.

### Viewing Receipt

The system should produce periodic viewing receipts: daily, weekly, and on demand.

They should answer:

- What was watched?
- What arrived that mattered?
- What was skipped?
- What did the curator flag?
- How did viewing patterns change?

Receipts should be calm, readable, configurable, archived in-repo, and valuable enough to support reflection rather than vanity metrics.

## YouTube API Surface (Strategic)

The station should be designed with explicit awareness of the highest-value YouTube API resources:

- `subscriptions`
- `channels`
- `playlistItems`
- `playlists`
- `videos` (details, statistics, `liveStreamingDetails`, player, `topicDetails`)
- `activities`
- `search`
- `commentThreads`
- `comments`
- `captions` (list and download)
- `videoCategories`
- `channelSections`
- `i18nLanguages`
- `i18nRegions`

### Quota Discipline

Daily quota is finite (default 10,000 units). The platform should treat quota as an editorial budget.

That means:

- prioritising subscription intelligence first
- spending expensive search calls intentionally
- using transcript and caption calls selectively
- preserving headroom for live checks and genuinely valuable user requests
- designing workflows so the system degrades gracefully when quota is constrained

## Data Model Ambition

The repository should become a high-integrity personal media dataset containing:

- subscription graph and channel metadata
- video records and temporal state
- playlist structure and ordering
- activity timelines
- live and premiere schedules
- transcript corpora and search indexes
- comment samples and community context
- curation tags and AI enrichments
- viewing logs and receipts
- historical guide snapshots

Interfaces can evolve. Layouts can be redesigned. The durable asset is the dataset, the history, and the editorial intelligence accumulated around them.

## Editorial Standard

Every feature should move the system toward:

- **intentional curation over maximal ingestion**
- **human readability over hidden complexity**
- **reliable automation over brittle cleverness**
- **product clarity over feature sprawl**
- **ownership over dependency**

## Standards for Future Work

Before implementing anything substantial, ask:

1. Does this make the station more personal?
2. Does this improve ownership, trust, and clarity?
3. Does this strengthen the data layer?
4. Does this preserve architectural boundaries?
5. Does this improve calmness, legibility, or usefulness?
6. Does this still work in multi-screen contexts?
7. Does this respect quota and cron-budget constraints?
8. Does this improve the AI curator's effectiveness?
9. Does this improve receipt quality and reflective value?
10. Does this use the right YouTube API surface for the job?
11. Does this materially improve the TV Guide, Channel Wall, or overall station quality?

If the answer is no, it is probably out of scope.

## Delivery Standard

To be considered viable, this repository should be able to demonstrate:

- reliable scheduled ingestion
- stable resource schemas
- fast static rendering
- genuinely useful guide and wall views
- transcript-aware discovery
- practical multi-screen coordination
- actionable conversational control through the personal YouTube AI Agent
- clear viewing receipts and historical accountability

Viability here does not mean novelty. It means the system is good enough, coherent enough, and dependable enough that someone would choose to use it repeatedly.

## Final Aspiration

I want this system to make YouTube utilisation excellent through ownership rather than addiction.

I want subscriptions to behave like channels.
I want playlists to behave like programme blocks.
I want the station to be programmable by conversation.
I want the data, history, and decisions to remain mine.

**The aspiration is a personal TV station in which YouTube provides the content, GitHub provides the infrastructure and memory, and a personal YouTube AI Agent provides the interface for thought, control, and curation.**
