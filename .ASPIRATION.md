# Aspiration

## I Love YouTube

My subscriptions feed is fabulous. It is a personally curated stream of creators, ideas, and content that reflects exactly what I care about. No algorithm fully controls it — I chose every channel, and the result is a feed that feels like mine.

That is the thing worth protecting here.

This project exists because the best part of YouTube is not YouTube's interface. The best part is the human curation I have already done: the channels I follow, the playlists I keep, the videos I like, and the patterns those choices reveal over time.

## The Core Thesis

I want to turn this GitHub repository into a **personal TV station** powered by my YouTube account.

Not a clone of YouTube. Not another social platform. Not an attempt to download and re-host the internet.

Something simpler and more personal:

- **YouTube remains the content library**
- **This repository becomes the processing and memory layer**
- **GitHub Pages becomes the screen**

The project is about taking a feed I already value and giving it a presentation layer I actually own.

## What This Repository Should Become

This repo should become a durable machine for turning YouTube activity into personal media infrastructure.

That means it should eventually do four jobs well:

1. **Collect** the parts of YouTube that matter to me.
2. **Normalize** them into structured, versioned resources.
3. **Curate** them into views that feel intentional rather than algorithmic.
4. **Present** them through a static site that is calm, legible, and mine.

If the project succeeds, opening the GitHub Pages site should feel less like opening an app and more like switching on a channel that was programmed for one specific person.

## The Architecture

### 1. Inside GitHub — Process YouTube into Resources

Using the **YouTube Data API** (authenticated via a `YOUTUBE_API_KEY` repository secret), GitHub Actions workflows will:

- **Pull subscriptions** — Retrieve the channels I subscribe to and the videos they publish.
- **Pull playlists** — Access curated playlists as programmable collections of content.
- **Pull liked videos** — Surface the videos I have explicitly endorsed.
- **Track metadata over time** — Keep enough structure to see when things changed, what is new, and what deserves attention.
- **Catalogue and index** — Transform raw API responses into stable JSON, Markdown, or other static resources committed to this repository.
- **Refresh on a schedule** — Run periodically so the repo reflects the current state of my YouTube world without requiring manual maintenance.
- **Process subscriptions in batches** — A cron job should be able to sweep through hundreds of channels on a regular cycle. If I follow a hundred channels, the system should update them in managed batches — not all at once, not one at a time, but in sensible groups that respect API rate limits and keep the data fresh. Think of it as the broadcast infrastructure refreshing its programme schedule overnight.
- **Support a Pay TV model** — Some channels deserve priority processing. The system should allow tagging or tiering subscriptions so that high-value channels are refreshed more frequently, while long-tail channels update on a slower cycle. This is cron-driven curation: the schedule itself becomes a statement of editorial priority.
- **Generate viewing receipts** — After a viewing session, or on a daily schedule, the system should compile and send an email receipt summarising what was watched, what was skipped, and what arrived new. The receipt treats attention like a transaction: something worth recording, reflecting on, and owning. GitHub Actions can assemble the receipt from viewing data tracked in the repository and deliver it via a configured email integration.

GitHub Actions is the **operations room**. The repository is the **archive**. Git is the **memory**.

### 2. Outside GitHub — A GitHub Page References YouTube

The **GitHub Pages** site published from this repository becomes the **frontend** — the actual personal TV station interface. It should:

- **Present subscriptions, playlists, and liked videos** in a way that is browsable and pleasant.
- **Link directly back to YouTube** for playback, respecting YouTube as the place where the videos actually live.
- **Stay static and fast** so it is cheap, durable, and easy to understand.
- **Expose curation choices clearly** so the page feels authored, not opaque.
- **Be configurable** through repo data, not hidden application state.
- **Work across multiple screens simultaneously** — Opening the GitHub Pages site on a laptop, a tablet, and a TV in the same room should produce a coordinated experience, not three identical copies. Screens should be able to adopt different roles: one shows the programme guide, another shows the active channel, a third monitors a wall of thumbnails. The site should use lightweight coordination (shared state via URL parameters, localStorage, or a broadcast channel) so that multiple browser tabs or devices can collaborate as a single viewing surface.
- **Present configurable channel walls** — The viewer should be able to see a grid of channels: 1×1 for focused viewing, 2×2 for a quick scan, 3×2 for a morning dashboard, all the way up to 10×10 for a full surveillance wall. The grid layout is a first-class presentation concept, not a hidden power-user feature. Channel walls turn subscriptions into something that feels like live television monitoring.

GitHub Pages is not just a place to dump data. It is the interface layer where taste becomes visible.

### 3. YouTube — The Source, Not the Owner of the Experience

YouTube remains the source of truth for the videos themselves, the channels, the playlists, and the interactions exposed by the API.

But YouTube should not be the sole owner of how I experience those things.

This project exists to separate **content source** from **consumption interface**.

## The Boundary

The architecture draws a clear line:

| Boundary | Role | What lives here |
|---|---|---|
| **Inside GitHub** (Actions, repo files) | Data processing and memory | API calls, transformations, schedules, snapshots, JSON/Markdown resources, curated metadata |
| **Cron layer** (GitHub Actions schedules) | Subscription lifecycle | Batch refresh cycles, priority tiers, Pay TV scheduling, rate-limit management |
| **Outside GitHub** (GitHub Pages) | Presentation | Static HTML/CSS/JS that reads processed data and links to YouTube |
| **Multi-screen coordination** | Distributed viewing | Screen roles, shared state, channel walls, layout negotiation between tabs/devices |
| **Intelligent curator** (AI layer) | Categorisation and conversation | AI-driven tagging, recommendation, natural-language channel management |
| **Outbound notifications** (email) | Viewing receipts and digests | Daily/weekly summaries, viewing receipts, attention reports, curator highlights |
| **YouTube** | Content source | Videos, channels, playlists, and platform-native playback |

YouTube is the **source**.  
This repository is the **processor**.  
GitHub Pages is the **screen**.

That separation matters because it keeps the system understandable. It avoids turning the repo into a bloated app while still making it powerful.

## What Makes This Different

This is not just a dashboard.

It is an attempt to treat personal media taste as something that deserves:

- **Ownership** — My viewing world should be expressible as data I control.
- **Transparency** — Every fetch, transformation, and publishing step should be inspectable in git.
- **Portability** — The useful layer is the structured data, not any one interface.
- **Composability** — Once the data exists in files, it can power feeds, schedules, digests, search, or entirely new interfaces.
- **Simplicity** — No app server, no database, no paid hosting, no unnecessary moving parts.
- **Longevity** — The system should survive because its pieces are plain, legible, and replaceable.

The point is not technical cleverness. The point is building a better relationship with a content stream I already care about.

## The Experience I Want

I do not want an infinite feed engineered for compulsion.

I want a system that helps me experience YouTube more deliberately:

- A clear sense of what is new from the channels I chose
- A readable view of playlists as intentional programming blocks
- A way to surface liked videos as trusted recommendations from my past self
- Rotating, themed, or time-based views that make the library feel alive
- The ability to feature some creators more prominently than others
- A homepage that feels closer to a program guide than a recommendation engine
- A channel wall that shows what is happening right now across every subscription at a glance
- The ability to open the station on multiple screens and have them work in concert — one screen as the guide, another as the player, another as the wall
- A conversation with an intelligent curator who knows my subscriptions, can suggest what to watch, and can build or reshape channel lineups on request
- An email receipt at the end of the day that tells me what I watched, what arrived, and what the curator thinks I should not miss — a paper trail for my attention that I actually own

The ideal outcome is that this station feels calmer than YouTube itself while still drawing from the same source material.

## The Product Shape

Over time, the personal TV station could include views such as:

- **Tonight's schedule** — a curated list of what to watch next
- **Latest from subscriptions** — recent uploads from chosen channels
- **Playlist blocks** — playlists treated like recurring shows or seasons
- **Favourites** — liked videos as a personal canon
- **Channel pages** — creator-specific views and archives
- **Themes or moods** — slices like learning, music, long-form, interviews, or comfort viewing
- **Archives and diffs** — what changed this week, what appeared, what disappeared
- **Channel wall** — a configurable N×M grid of live channel tiles, from 1×1 single focus to 10×10 full surveillance, each tile showing the latest or currently playing content from a subscription
- **Multi-screen views** — companion layouts where one screen acts as a remote control or programme guide while another screen plays, or where a wall of screens each takes a different grid segment
- **Curator chat** — a conversational interface where you ask for channels by mood, topic, or creator, and the system builds a lineup, opens tiles, or reshapes the wall in response
- **AI-generated schedules** — programming blocks assembled by an intelligent agent that understands your viewing history, categorises content, and proposes what to watch when
- **Viewing receipts** — daily or weekly email digests that summarise what was watched, what arrived new, what the curator highlighted, and how viewing time was spent — a personal statement of account for attention

These are examples, not commitments. The real commitment is that every feature should make the station feel more intentional, more personal, and more legible.

## The Cron Processing Layer

The station is only as good as its data freshness. A personal TV station with stale programme listings is just a static archive.

GitHub Actions cron schedules should power a **subscription processing pipeline** that keeps the repository current:

- **Batch processing** — If a hundred channels are subscribed, the system should not try to refresh them all in a single run. It should process them in configurable batches (say, 20 at a time across five scheduled runs), cycling through the full subscription list over the course of hours or a day.
- **Priority tiers** — Not all channels are equal. A "Pay TV" tier of high-value channels should refresh every hour. A standard tier refreshes every six hours. A long-tail tier refreshes daily. The cron schedule becomes an expression of editorial priority.
- **Rate-limit awareness** — The YouTube Data API has daily quota limits. The processing pipeline must track quota usage, throttle requests when limits approach, and distribute work across cron windows to maximise coverage without hitting walls.
- **Incremental updates** — Each cron run should commit only what changed. New videos appear as new entries. Removed videos are marked, not deleted. Metadata updates are diffed. Git history becomes the changelog of the entire subscription world.
- **Health monitoring** — The system should track which channels were last refreshed and surface staleness. If a channel has not been updated in 48 hours, that is visible. If a cron run fails, the next run knows where to resume.

The aspiration is a cron layer that runs quietly in the background, keeping the station's programme data fresh the way a broadcast engineer keeps the signal live — invisible when working, immediately obvious when it stops.

## The Channel Wall

A channel wall is the visual centrepiece of the station. It is what makes the experience feel like television rather than a website.

The wall is an N×M grid of channel tiles, where each tile represents one subscription or curated channel. The grid is configurable:

| Layout | Feel | Use case |
|---|---|---|
| **1×1** | Single focused channel | Watching one thing with full attention |
| **2×1** or **1×2** | Side-by-side or stacked pair | Comparing two channels, picture-in-picture |
| **2×2** | Compact quad view | Morning dashboard, checking four favourites |
| **3×3** | Nine-channel grid | A programme block or themed cluster |
| **4×3** | Twelve-channel wall | A category overview — all music, all news, all tech |
| **5×5** | Twenty-five channels | Full subscription scan |
| **10×10** | Hundred-channel wall | Total surveillance mode — every subscription visible at once |

Each tile in the wall should show:

- The channel name and avatar
- The most recent or currently featured video thumbnail
- A visual indicator of freshness (new content, updated today, stale)
- A click or tap to expand that tile to full screen or to a larger segment of the grid

The wall should be **responsive** — it adapts to the screen size, so a 10×10 grid on a 4K TV shows rich thumbnails while the same grid on a phone collapses to a scrollable list.

The wall should be **configurable** — saved layouts stored as JSON in the repository define which channels appear in which positions. Different walls for different moods, times of day, or viewing contexts.

The wall should be **alive** — tiles should rotate or update as cron processing delivers fresh data. The wall is not a snapshot; it is a living mosaic of the subscription world.

## The Multi-Screen Experience

A personal TV station should not be limited to a single browser tab. The aspiration is that the GitHub Pages site can be opened on multiple screens — a laptop, a tablet, a phone, a TV — and those screens **work together** as a unified viewing surface.

This means:

- **Screen roles** — Each screen can adopt a role: controller, player, wall, guide, or curator chat. A phone becomes the remote control. A TV becomes the player. A tablet shows the channel wall. A laptop hosts the curator conversation.
- **Lightweight coordination** — Screens in the same session share state through a simple mechanism: URL fragment parameters, `BroadcastChannel` API for same-device tabs, or `localStorage` polling for cross-tab sync. No server required. No WebSocket infrastructure. The coordination layer must be as static and durable as the rest of the system.
- **Independent operation** — Every screen must also work independently. If the phone loses connection to the session, it still functions as a standalone station. Multi-screen is an enhancement, not a dependency.
- **Layout distribution** — A 10×10 channel wall could be distributed across four screens, each showing a 5×5 quadrant. Or one screen shows the wall while another shows the expanded view of whatever tile was selected. The system should support splitting the wall across physical screens.
- **Session awareness** — Opening the site on a new device should offer the option to join an existing session or start a fresh one. Sessions are identified by a simple shared key, not by authentication.

The multi-screen experience should feel like walking into a control room where every display serves a purpose, and adding a new screen makes the room more capable, not more cluttered.

## The Intelligent Curator

This is the 24th-century aspiration. The station should eventually include an **AI-powered digital curator** that understands the subscription world and can participate in managing it.

The curator is not a recommendation algorithm. It is a conversational partner:

- **AI categorisation** — The curator automatically tags and categorises channels and videos: genre, topic, mood, format, length, language, production style. These categories are stored as structured metadata in the repository, enriching the data model far beyond what the YouTube API provides.
- **Chat interface** — The viewer can talk to the curator through a chat panel on the station. "Show me something calming." "What is new from my science channels?" "Build me a 3×3 wall of music channels." "What have I been watching too much of lately?" The curator responds with actions: reshaping the wall, queuing videos, surfacing channels, or offering reflections.
- **Channel management through conversation** — Instead of manually editing JSON or navigating settings, the viewer tells the curator what they want. "Add this channel to my morning wall." "Move cooking channels to the weekend schedule." "Drop channels I have not watched in six months." The curator translates natural language into data changes committed to the repository.
- **Subscription monitoring** — The cron layer feeds the curator. As new videos arrive, the curator can flag notable content, detect patterns (a creator has not posted in weeks, a channel changed its focus, a topic is trending across multiple subscriptions), and surface insights proactively.
- **Schedule generation** — The curator can propose viewing schedules: "Tonight's picks based on what arrived today," or "A weekend marathon of your liked videos from the past month." These schedules are generated as structured data that the presentation layer can render as a programme guide.
- **Memory and learning** — The curator's understanding improves over time because every interaction is stored in the repository. It can reference past conversations, recall preferences stated weeks ago, and adapt its suggestions to evolving taste. Git is the memory. The repository is the mind.

The curator should feel like a knowledgeable librarian who has read every video title, description, and comment in the subscription world, and who is always available to help navigate it. Not pushy. Not manipulative. Just deeply informed and genuinely helpful.

The intelligent curator is where the GitHub Minimum Intelligence agent framework and the personal TV station converge: the same repository that hosts the station also hosts the AI that curates it.

## The Viewing Receipt

When you spend money, you get a receipt. When you spend attention, you get nothing. The viewing receipt changes that.

The idea is simple: the station should send you an **email receipt** that accounts for your viewing activity the way a bank statement accounts for your spending. Not to guilt or gamify — but to make consumption visible, reflective, and owned.

A viewing receipt could arrive daily, weekly, or after each session. It might include:

- **What you watched** — The videos you opened, the channels you visited, the time you spent. A clear, honest record of where your attention went.
- **What arrived new** — A summary of fresh content from your subscriptions since the last receipt. The cron layer already tracks this; the receipt simply surfaces it in your inbox.
- **What the curator flagged** — Highlights, recommendations, or insights from the intelligent curator. "Three of your science channels posted this week." "A creator you follow just started a new series." "You have not watched anything from your music subscriptions in two weeks."
- **Viewing patterns** — Over time, receipts accumulate into a personal viewing history. Weekly totals, favourite channels this month, genres that dominate, creators you are drifting away from. Not analytics designed to maximise engagement — analytics designed to help you understand your own habits.
- **What you skipped** — The receipt could note what was available but unwatched, giving a gentle sense of what the station offered versus what was actually consumed.

The receipt should feel like a **letter from your station**, not a notification from an app. It should be:

- **Calm and readable** — Plain text or simple HTML. No tracking pixels, no click-bait subject lines, no dark patterns.
- **Configurable** — Choose daily, weekly, or on-demand. Choose what sections to include. Choose whether to receive it at all.
- **Generated from repo data** — The receipt is assembled by a GitHub Actions workflow that reads the viewing log and subscription state from the repository, formats it, and sends it via a configured email integration (SendGrid, SMTP, or GitHub's own notification system).
- **Archived in the repository** — Each receipt is also committed as a Markdown file, creating a permanent, searchable history of viewing summaries. The inbox copy is ephemeral; the repo copy is forever.

The deeper point is philosophical. Platforms track everything you watch but never give that data back in a form that serves you. YouTube knows your viewing history better than you do, but it uses that knowledge to optimise its own goals. The viewing receipt reclaims that knowledge. It says: this is what I chose to watch, on my terms, summarised in my format, delivered to my inbox, and archived in my repository.

Attention deserves a paper trail. The viewing receipt provides one.

## The Data Model Aspiration

The most important asset this project can produce is not the website. It is the dataset.

The repo should gradually become a high-quality personal record of:

- Which channels I subscribe to
- Which videos those channels publish
- Which playlists I maintain or care about
- Which videos I have liked
- Which items are new, recurring, featured, archived, or manually highlighted
- Which videos were actually watched, when, and for how long — the viewing log that powers receipts and reflection
- Which viewing receipts were sent, and what patterns they revealed over time

Good structured data makes everything else possible. If the underlying resources are clean, stable, and versioned, then the interface can evolve freely without losing the real value.

## The Editorial Standard

The station should feel authored.

That means the project should prefer:

- **Intentional curation over maximal ingestion**
- **Readable pages over clever interfaces**
- **Stable structures over brittle automation**
- **Human legibility over hidden magic**

The repository should not become a generic mirror of everything YouTube exposes. It should become a selective, expressive representation of the parts that matter.

## The Philosophical Point

This is not just a tool. It is a statement about how content consumption could work differently.

Today, platforms give us feeds but own the experience. They choose the layout, the ranking, the defaults, the surrounding noise, and the behavioral incentives. Even when the underlying content is excellent, the interface is designed for the platform's goals first.

This project argues for a different model:

- Let the platform host the content.
- Let the user own the organisation.
- Let the pipeline be inspectable.
- Let the interface be reshapeable.
- Let taste become infrastructure.

Even if this only serves one person, it is still meaningful. A personal system can be a real alternative, not just a prototype.

## The Standard for Future Work

Before building any feature, it should be possible to ask:

1. Does this make the station more personal?
2. Does this increase ownership or clarity?
3. Does this improve the quality of the data layer?
4. Does this keep the boundary between source, processing, and presentation clean?
5. Does this make the experience calmer, clearer, or more intentional?
6. Does this work when the station is displayed across multiple screens?
7. Does this respect the cron processing budget and API rate limits?
8. Does this make the intelligent curator smarter or more helpful?
9. Does this contribute to a meaningful viewing receipt or help the viewer reflect on their habits?

If the answer is no, the feature is probably a distraction.
