# Aspiration

## I Love YouTube

My subscriptions feed is fabulous. It is a personally curated stream of creators, ideas, and content that reflects exactly what I care about. No algorithm fully controls it — I chose every channel, and the result is a feed that feels like mine.

That is the thing worth protecting here.

This project exists because the best part of YouTube is not YouTube's interface. The best part is the human curation I have already done: the channels I follow, the playlists I keep, the videos I like, and the patterns those choices reveal over time.

## The Core Thesis

I want to turn this GitHub repository into a **personal TV station** powered by my YouTube account.

Not a clone of YouTube. Not another social platform. Not an attempt to download and re-host the internet.

Something simpler and more personal:

- **YouTube remains the content library**
- **This repository becomes the processing and memory layer**
- **GitHub Pages becomes the screen**

The project is about taking a feed I already value and giving it a presentation layer I actually own.

## What This Repository Should Become

This repo should become a durable machine for turning YouTube activity into personal media infrastructure.

That means it should eventually do four jobs well:

1. **Collect** the parts of YouTube that matter to me.
2. **Normalize** them into structured, versioned resources.
3. **Curate** them into views that feel intentional rather than algorithmic.
4. **Present** them through a static site that is calm, legible, and mine.

If the project succeeds, opening the GitHub Pages site should feel less like opening an app and more like switching on a channel that was programmed for one specific person.

## The Architecture

### 1. Inside GitHub — Process YouTube into Resources

Using the **YouTube Data API** (authenticated via a `YOUTUBE_API_KEY` repository secret, and where needed an OAuth 2.0 token for private data), GitHub Actions workflows will:

- **Pull subscriptions** — Retrieve the channels I subscribe to and the videos they publish.
- **Pull playlists** — Access curated playlists as programmable collections of content.
- **Pull liked videos** — Surface the videos I have explicitly endorsed.
- **Pull channel details** — Retrieve rich metadata for every subscribed channel: description, subscriber count, total views, video count, topic categories, featured channels, branding, and upload playlist ID. This data powers tiering, the channel wall, and the curator's understanding of each source.
- **Pull video details** — For each video, fetch not just the title and thumbnail but the full resource: duration, definition (HD/SD), caption availability, embeddability, content ratings, region restrictions, tags, category, language, topic categories, recording location, and live streaming status. These details make filtering, sorting, and intelligent presentation possible.
- **Pull video statistics** — Retrieve viewCount, likeCount, and commentCount for videos. Engagement data is a curation signal: it tells the curator what is resonating, what is being discussed, and what the world thinks is worth watching — without letting it become the only signal.
- **Pull activity feeds** — Use the Activities resource to track what subscribed channels have been doing: uploads, likes, favourites, playlist additions, and new subscriptions. The activity feed is a timeline of the subscription world's behaviour, not just its latest output.
- **Pull comments** — Retrieve comment threads for videos. Comments are community intelligence: they reveal discussion, controversy, appreciation, and context that metadata alone cannot capture. Selectively archiving top-level comments enriches the station's understanding of its content.
- **Pull captions and transcripts** — List available caption tracks for videos and, where access allows, download them. Transcripts turn video into text. Text is searchable, summarisable, and analysable. A station that knows what was said in a video — not just what it was titled — is a station that can find, recommend, and reflect on content at a fundamentally deeper level.
- **Search the YouTube catalogue** — Use the Search resource to find videos, channels, and playlists by keyword, topic, location, date range, duration, definition, caption availability, event type, and more. Search is how the station discovers content beyond the subscription list — and how the curator answers questions like "find me a recent long-form interview about climate science."
- **Fetch video categories and reference data** — Retrieve YouTube's own video category taxonomy, supported languages, and supported regions. These provide a stable classification backbone that the AI curator can build on rather than starting from scratch.
- **Detect live content** — Use live streaming details to identify which videos are live right now, which are upcoming, and which have ended. For a TV station, live awareness is fundamental: the channel wall should be able to highlight a channel that is broadcasting live, and the programme guide should surface upcoming premieres and scheduled streams.
- **Track metadata over time** — Keep enough structure to see when things changed, what is new, and what deserves attention.
- **Catalogue and index** — Transform raw API responses into stable JSON, Markdown, or other static resources committed to this repository.
- **Refresh on a schedule** — Run periodically so the repo reflects the current state of my YouTube world without requiring manual maintenance.
- **Process subscriptions in batches** — A cron job should be able to sweep through hundreds of channels on a regular cycle. If I follow a hundred channels, the system should update them in managed batches — not all at once, not one at a time, but in sensible groups that respect API rate limits and keep the data fresh. Think of it as the broadcast infrastructure refreshing its programme schedule overnight.
- **Support a Pay TV model** — Some channels deserve priority processing. The system should allow tagging or tiering subscriptions so that high-value channels are refreshed more frequently, while long-tail channels update on a slower cycle. This is cron-driven curation: the schedule itself becomes a statement of editorial priority.
- **Generate viewing receipts** — After a viewing session, or on a daily schedule, the system should compile and send an email receipt summarising what was watched, what was skipped, and what arrived new. The receipt treats attention like a transaction: something worth recording, reflecting on, and owning. GitHub Actions can assemble the receipt from viewing data tracked in the repository and deliver it via a configured email integration.

GitHub Actions is the **operations room**. The repository is the **archive**. Git is the **memory**.

### 2. Outside GitHub — A GitHub Page References YouTube

The **GitHub Pages** site published from this repository becomes the **frontend** — the actual personal TV station interface. It should:

- **Present subscriptions, playlists, and liked videos** in a way that is browsable and pleasant.
- **Link directly back to YouTube** for playback, respecting YouTube as the place where the videos actually live.
- **Embed where it enriches the experience** — The YouTube API provides embed HTML for every embeddable video. The station can use this to offer inline playback for channel wall tiles, preview panes, or focused viewing — keeping the viewer on the station while still drawing from YouTube. Embedding is optional and respectful: it uses YouTube's own player, complies with its terms, and falls back to a link when a video is not embeddable.
- **Surface live and upcoming content** — When a subscribed channel is broadcasting live or has a premiere scheduled, the station should make that visible: a live badge on the channel wall, a countdown for upcoming streams, a prominent position in the programme guide. The API provides this data; the station should use it.
- **Offer transcript search** — If captions have been archived for videos, the station should let the viewer search within them. "Find the part where they talk about quantum computing" becomes a real query when the station has the transcript text. This is a superpower that YouTube itself does not expose in its own interface.
- **Show engagement context** — View counts, like counts, and comment counts can be presented as quiet contextual signals: a small indicator of how widely watched or discussed a video is. Not as a ranking mechanism, but as information that helps the viewer make an informed choice.
- **Stay static and fast** so it is cheap, durable, and easy to understand.
- **Expose curation choices clearly** so the page feels authored, not opaque.
- **Be configurable** through repo data, not hidden application state.
- **Work across multiple screens simultaneously** — Opening the GitHub Pages site on a laptop, a tablet, and a TV in the same room should produce a coordinated experience, not three identical copies. Screens should be able to adopt different roles: one shows the programme guide, another shows the active channel, a third monitors a wall of thumbnails. The site should use lightweight coordination (shared state via URL parameters, localStorage, or a broadcast channel) so that multiple browser tabs or devices can collaborate as a single viewing surface.
- **Present configurable channel walls** — The viewer should be able to see a grid of channels: 1×1 for focused viewing, 2×2 for a quick scan, 3×2 for a morning dashboard, all the way up to 10×10 for a full surveillance wall. The grid layout is a first-class presentation concept, not a hidden power-user feature. Channel walls turn subscriptions into something that feels like live television monitoring.

GitHub Pages is not just a place to dump data. It is the interface layer where taste becomes visible.

### 3. YouTube — The Source, Not the Owner of the Experience

YouTube remains the source of truth for the videos themselves, the channels, the playlists, and the interactions exposed by the API.

But YouTube should not be the sole owner of how I experience those things.

This project exists to separate **content source** from **consumption interface**.

## The Boundary

The architecture draws a clear line:

| Boundary | Role | What lives here |
|---|---|---|
| **Inside GitHub** (Actions, repo files) | Data processing and memory | API calls, transformations, schedules, snapshots, JSON/Markdown resources, curated metadata, archived transcripts, comment summaries |
| **Cron layer** (GitHub Actions schedules) | Subscription lifecycle | Batch refresh cycles, priority tiers, Pay TV scheduling, quota-aware rate-limit management, activity feed polling |
| **Outside GitHub** (GitHub Pages) | Presentation | Static HTML/CSS/JS that reads processed data, embeds YouTube playback, surfaces transcripts, and links to YouTube |
| **TV guide** (programme listings) | Schedule and discovery | Time-based grid, upcoming premieres, live-now indicators, genre rows, editorial blocks, search-driven discovery |
| **Multi-screen coordination** | Distributed viewing | Screen roles, shared state, channel walls, layout negotiation between tabs/devices |
| **Intelligent curator** (AI layer) | Categorisation and conversation | AI-driven tagging, recommendation, natural-language channel management, transcript analysis, comment sentiment |
| **Outbound notifications** (email) | Viewing receipts and digests | Daily/weekly summaries, viewing receipts, attention reports, curator highlights |
| **YouTube** | Content source and API surface | Videos, channels, playlists, captions, comments, activities, categories, search, live streams, and platform-native playback |

YouTube is the **source**.  
This repository is the **processor**.  
GitHub Pages is the **screen**.

That separation matters because it keeps the system understandable. It avoids turning the repo into a bloated app while still making it powerful.

## What Makes This Different

This is not just a dashboard.

It is an attempt to treat personal media taste as something that deserves:

- **Ownership** — My viewing world should be expressible as data I control.
- **Transparency** — Every fetch, transformation, and publishing step should be inspectable in git.
- **Portability** — The useful layer is the structured data, not any one interface.
- **Composability** — Once the data exists in files, it can power feeds, schedules, digests, search, or entirely new interfaces.
- **Simplicity** — No app server, no database, no paid hosting, no unnecessary moving parts.
- **Longevity** — The system should survive because its pieces are plain, legible, and replaceable.

The point is not technical cleverness. The point is building a better relationship with a content stream I already care about.

## The Product Shape

Over time, the personal TV station could include views such as:

- **The TV guide** — a full electronic programme guide: a time-based grid showing what is on now, what is coming next, and what aired recently across every subscription. Channels run down the left, time runs across the top, and each cell is a video. The guide should feel like opening a newspaper's television listings or switching to the EPG on a set-top box — except every channel is one you chose, and the schedule is assembled from real API data rather than a broadcast signal.
- **Tonight's schedule** — a curated list of what to watch next
- **Latest from subscriptions** — recent uploads from chosen channels
- **Live now** — a view that highlights channels currently broadcasting live, with a live badge, concurrent viewer count, and a one-click path to the stream. The API's live streaming details make this possible. A TV station that does not know what is live is not really a TV station.
- **Upcoming premieres** — scheduled live streams and premieres surfaced from the API, displayed as a countdown calendar or upcoming-events strip. The station should know what is about to happen, not just what already happened.
- **Playlist blocks** — playlists treated like recurring shows or seasons
- **Favourites** — liked videos as a personal canon
- **Channel pages** — creator-specific views and archives, enriched with channel statistics (subscriber count, video count, total views), topic categories, featured channels, and recent activity
- **Themes or moods** — slices like learning, music, long-form, interviews, or comfort viewing — powered by YouTube's video categories, topic details, tags, and AI-enhanced tagging from the curator
- **Duration filters** — views filtered by video length: short-form (under 4 minutes), medium (4–20 minutes), or long-form (over 20 minutes). The API provides duration data for every video. Sometimes you have five minutes; sometimes you have an evening. The station should know the difference.
- **Transcript search** — a search interface that queries archived caption text across the video library. "Find me the episode where they discuss supply chains" becomes a real query when the station has transcripts. This is a capability YouTube itself does not offer its viewers.
- **Comment highlights** — surfaced comment threads for videos, showing what the community is saying. Not a full comment section, but a curated excerpt: the top discussion points, the most-liked replies, the creator's own responses. Comments are context that makes watching more informed.
- **Discovery search** — a search panel powered by the YouTube Search API that lets the viewer find content beyond their subscriptions: by keyword, topic, channel, date, duration, or language. Discovery is how the subscription list grows. The station should support it.
- **Activity timeline** — a chronological feed of what subscribed channels have been doing: uploads, likes, playlist additions, new subscriptions. The activity feed tells a story about each channel's behaviour over time, not just its latest video.
- **Archives and diffs** — what changed this week, what appeared, what disappeared
- **Channel wall** — a configurable N×M grid of live channel tiles, from 1×1 single focus to 10×10 full surveillance, each tile showing the latest or currently playing content from a subscription
- **Multi-screen views** — companion layouts where one screen acts as a remote control or programme guide while another screen plays, or where a wall of screens each takes a different grid segment
- **Curator chat** — a conversational interface where you ask for channels by mood, topic, or creator, and the system builds a lineup, opens tiles, or reshapes the wall in response
- **AI-generated schedules** — programming blocks assembled by an intelligent agent that understands your viewing history, categorises content, and proposes what to watch when
- **Viewing receipts** — daily or weekly email digests that summarise what was watched, what arrived new, what the curator highlighted, and how viewing time was spent — a personal statement of account for attention

These are examples, not commitments. The real commitment is that every feature should make the station feel more intentional, more personal, and more legible.

## The Cron Processing Layer

The station is only as good as its data freshness. A personal TV station with stale programme listings is just a static archive.

GitHub Actions cron schedules should power a **subscription processing pipeline** that keeps the repository current:

- **Batch processing** — If a hundred channels are subscribed, the system should not try to refresh them all in a single run. It should process them in configurable batches (say, 20 at a time across five scheduled runs), cycling through the full subscription list over the course of hours or a day.
- **Priority tiers** — Not all channels are equal. A "Pay TV" tier of high-value channels should refresh every hour. A standard tier refreshes every six hours. A long-tail tier refreshes daily. The cron schedule becomes an expression of editorial priority.
- **Rate-limit awareness** — The YouTube Data API has daily quota limits. The default allocation is 10,000 units per day. Not all API calls cost the same: listing videos, channels, or playlists costs 1 unit per call, but a search costs 100 units. Caption operations cost 50 units. The processing pipeline must track quota usage, understand the per-method cost structure, throttle requests when limits approach, and distribute work across cron windows to maximise coverage without hitting walls. The quota budget is a hard constraint that shapes what the station can afford to know and how often.
- **Incremental updates** — Each cron run should commit only what changed. New videos appear as new entries. Removed videos are marked, not deleted. Metadata updates are diffed. Git history becomes the changelog of the entire subscription world.
- **Health monitoring** — The system should track which channels were last refreshed and surface staleness. If a channel has not been updated in 48 hours, that is visible. If a cron run fails, the next run knows where to resume.
- **Transcript harvesting** — On a slower cycle, the cron layer should check for available caption tracks on recently ingested videos and archive transcript text where accessible. Not every video has captions, and not every caption track is downloadable, but those that are represent an extraordinary enrichment of the data model. Transcripts should be harvested opportunistically, respecting both quota and access constraints.
- **Comment sampling** — Periodically, the cron layer should retrieve top-level comment threads for high-priority or recently watched videos. Full comment archives are neither practical nor desirable, but a sample of community discussion — especially for videos the viewer has engaged with — enriches the station's intelligence without exhausting the quota.
- **Activity polling** — The activity feeds of subscribed channels should be polled on a regular cycle. Activities are lightweight (1 unit per call) and provide a timeline of channel behaviour that complements the video-by-video refresh: new uploads, new subscriptions, playlist changes. The activity feed is the station's awareness of what is happening across its subscription world right now.
- **Live content detection** — Each cron run should check for live broadcasts and upcoming premieres among subscribed channels. Live awareness is time-sensitive: a live badge that appears an hour after the stream ended is useless. The cron schedule should allocate a frequent, lightweight check for live status on high-priority channels.

The aspiration is a cron layer that runs quietly in the background, keeping the station's programme data fresh the way a broadcast engineer keeps the signal live — invisible when working, immediately obvious when it stops.

## The Channel Wall

A channel wall is the visual centrepiece of the station. It is what makes the experience feel like television rather than a website.

The wall is an N×M grid of channel tiles, where each tile represents one subscription or curated channel. The grid is configurable:

| Layout | Feel | Use case |
|---|---|---|
| **1×1** | Single focused channel | Watching one thing with full attention |
| **2×1** or **1×2** | Side-by-side or stacked pair | Comparing two channels, picture-in-picture |
| **2×2** | Compact quad view | Morning dashboard, checking four favourites |
| **3×3** | Nine-channel grid | A programme block or themed cluster |
| **4×3** | Twelve-channel wall | A category overview — all music, all news, all tech |
| **5×5** | Twenty-five channels | Full subscription scan |
| **10×10** | Hundred-channel wall | Total surveillance mode — every subscription visible at once |

Each tile in the wall should show:

- The channel name and avatar
- The most recent or currently featured video thumbnail
- A visual indicator of freshness (new content, updated today, stale)
- A **live badge** when the channel is currently broadcasting, with concurrent viewer count — because a TV wall that does not know what is live is just a poster wall
- Video duration and definition (HD/SD) as quiet metadata
- A click or tap to expand that tile to full screen or to a larger segment of the grid

The wall should be **responsive** — it adapts to the screen size, so a 10×10 grid on a 4K TV shows rich thumbnails while the same grid on a phone collapses to a scrollable list.

The wall should be **configurable** — saved layouts stored as JSON in the repository define which channels appear in which positions. Different walls for different moods, times of day, or viewing contexts.

The wall should be **alive** — tiles should rotate or update as cron processing delivers fresh data. The wall is not a snapshot; it is a living mosaic of the subscription world.

## The TV Guide

The channel wall shows what is happening at a glance. The TV guide shows what is happening over time.

A full electronic programme guide is the missing piece that turns a collection of videos into something that feels like broadcast television. The TV guide is a **time-based grid** — channels down the left, time across the top — that organises the subscription world into a browsable schedule.

YouTube does not broadcast on a fixed schedule the way traditional television does. But it does have temporal structure: videos are published at specific times, live streams are scheduled for specific moments, premieres count down to a specific start. The cron layer already tracks when content arrives. The TV guide makes that temporal structure visible and navigable.

### What the guide shows

- **Now** — What was most recently published by each channel, and which channels are currently live. The "now" column is the heartbeat of the guide: it answers the question "what is on?" the same way a set-top box EPG does.
- **Next** — Upcoming premieres, scheduled live streams, and the predicted next upload for channels with regular publishing cadences. The API provides `scheduledStartTime` for live broadcasts and premieres; the cron layer can infer publishing patterns from historical data.
- **Recent** — A scrollable timeline of what has already aired: the past 24 hours, the past week, the past month. Each cell is a video with its title, thumbnail, duration, and engagement indicators (view count, comment count).
- **Live strip** — A prominent row or badge system that highlights every channel currently broadcasting live, with concurrent viewer counts and a direct path to the stream. Live content gets priority placement because it is ephemeral: miss it now and it becomes a recording.
- **Premiere countdown** — Upcoming premieres displayed with countdown timers. The API's `liveStreamingDetails.scheduledStartTime` powers this directly.

### Guide layouts

| Layout | Feel | Use case |
|---|---|---|
| **Classic grid** | Channels × time, like a cable EPG | Full programme overview, browsing what is on |
| **Genre rows** | Grouped by category or topic | "Show me all the science channels," "What music is on?" |
| **Single channel** | Detailed timeline for one channel | Deep dive into a creator's recent output |
| **Editorial blocks** | Curator-assembled time slots | "Morning briefing," "Friday night movies," "Weekend deep dives" |
| **Highlights strip** | Horizontal scroll of the guide's top picks | A quick-glance summary for someone who just wants a recommendation |

### What makes it a real guide

- **Powered by API data** — Every cell in the guide is backed by real data from the YouTube API: video resources, live streaming details, activity feeds, and channel metadata. The guide does not guess; it knows.
- **Enriched by the curator** — The AI curator can tag videos with genres, moods, and quality assessments that YouTube's own categories do not capture. These enrichments make the genre rows and editorial blocks possible.
- **Searchable** — The guide should support filtering by keyword, duration, category, topic, language, and whether captions are available. If the station has archived transcripts, the guide can even search within what was said in videos.
- **Time-zone aware** — The guide should present times in the viewer's local time zone, not UTC. Publishing times from the API are UTC; the presentation layer converts them.
- **Configurable density** — The viewer should be able to choose how much detail each cell shows: thumbnail only, thumbnail plus title, full metadata with statistics, or a compact text-only listing. The guide should work at a glance and also reward closer inspection.
- **Linked to the wall** — Selecting a channel in the guide should be able to tune the channel wall to that channel. The guide and the wall are two views of the same data, and they should be connected.
- **Multi-screen role** — On a multi-screen setup, one screen can display the TV guide while another displays the channel wall or the player. The guide becomes the remote control: browse the schedule on the tablet, select a programme, and it plays on the TV.

The TV guide is the anchor of the station's identity. Without it, the station is a collection of feeds. With it, the station is a place with a schedule, a sense of time, and a rhythm that the viewer can learn and rely on.

The aspiration is that opening the TV guide on a Sunday morning should feel like unfolding a newspaper's television section: calm, complete, and full of possibility.

## The Multi-Screen Experience

A personal TV station should not be limited to a single browser tab. The aspiration is that the GitHub Pages site can be opened on multiple screens — a laptop, a tablet, a phone, a TV — and those screens **work together** as a unified viewing surface.

This means:

- **Screen roles** — Each screen can adopt a role: controller, player, wall, guide, or curator chat. A phone becomes the remote control. A TV becomes the player. A tablet shows the channel wall. A laptop hosts the curator conversation.
- **Lightweight coordination** — Screens in the same session share state through a simple mechanism: URL fragment parameters, `BroadcastChannel` API for same-device tabs, or `localStorage` polling for cross-tab sync. No server required. No WebSocket infrastructure. The coordination layer must be as static and durable as the rest of the system.
- **Independent operation** — Every screen must also work independently. If the phone loses connection to the session, it still functions as a standalone station. Multi-screen is an enhancement, not a dependency.
- **Layout distribution** — A 10×10 channel wall could be distributed across four screens, each showing a 5×5 quadrant. Or one screen shows the wall while another shows the expanded view of whatever tile was selected. The system should support splitting the wall across physical screens.
- **Session awareness** — Opening the site on a new device should offer the option to join an existing session or start a fresh one. Sessions are identified by a simple shared key, not by authentication.

The multi-screen experience should feel like walking into a control room where every display serves a purpose, and adding a new screen makes the room more capable, not more cluttered.

## The Intelligent Curator

This is the 24th-century aspiration. The station should eventually include an **AI-powered digital curator** that understands the subscription world and can participate in managing it.

The curator is not a recommendation algorithm. It is a conversational partner:

- **AI categorisation** — The curator automatically tags and categorises channels and videos: genre, topic, mood, format, length, language, production style. It starts with what the YouTube API already provides — video categories, topic details linked to Wikipedia, channel topic categories, tags — and builds richer, more personal categories on top. YouTube's taxonomy is a foundation; the curator's taxonomy is the editorial layer.
- **Transcript intelligence** — When transcripts are available, the curator can read them. It can summarise what a video is about, extract key topics, identify recurring themes across a creator's output, and answer natural-language questions about content. "What did this channel say about AI last month?" becomes answerable when the curator has the text.
- **Comment awareness** — The curator can read archived comment samples and use them as a signal of community sentiment: what the audience loved, what was controversial, what sparked discussion. Comments are noisy, but a curator that has never read the room is a curator operating blind.
- **Chat interface** — The viewer can talk to the curator through a chat panel on the station. "Show me something calming." "What is new from my science channels?" "Build me a 3×3 wall of music channels." "What have I been watching too much of lately?" The curator responds with actions: reshaping the wall, queuing videos, surfacing channels, or offering reflections.
- **Search delegation** — The curator can use the YouTube Search API on the viewer's behalf. "Find me a recent documentary about ocean currents" triggers a search, and the curator presents the results within the station's interface, filtered and ranked by its understanding of the viewer's preferences. Search costs 100 quota units per call, so the curator should be judicious — but when the viewer asks to discover, the curator should be able to look beyond the subscription list.
- **Channel management through conversation** — Instead of manually editing JSON or navigating settings, the viewer tells the curator what they want. "Add this channel to my morning wall." "Move cooking channels to the weekend schedule." "Drop channels I have not watched in six months." The curator translates natural language into data changes committed to the repository.
- **TV guide programming** — The curator can assemble and populate the TV guide's editorial blocks. "Programme a Sunday morning lineup of calm, long-form content." "Fill the science row with this week's best uploads." The guide is the canvas; the curator is the programmer.
- **Subscription monitoring** — The cron layer feeds the curator. As new videos arrive, the curator can flag notable content, detect patterns (a creator has not posted in weeks, a channel changed its focus, a topic is trending across multiple subscriptions), and surface insights proactively. Activity feeds and channel statistics enrich this monitoring: the curator notices when a channel's upload frequency changes, when subscriber counts shift, or when a channel starts featuring new collaborators.
- **Schedule generation** — The curator can propose viewing schedules: "Tonight's picks based on what arrived today," or "A weekend marathon of your liked videos from the past month." These schedules are generated as structured data that the presentation layer can render as a programme guide.
- **Memory and learning** — The curator's understanding improves over time because every interaction is stored in the repository. It can reference past conversations, recall preferences stated weeks ago, and adapt its suggestions to evolving taste. Git is the memory. The repository is the mind.

The curator should feel like a knowledgeable librarian who has read every video title, description, and comment in the subscription world, and who is always available to help navigate it. Not pushy. Not manipulative. Just deeply informed and genuinely helpful.

The intelligent curator is where the GitHub Minimum Intelligence agent framework and the personal TV station converge: the same repository that hosts the station also hosts the AI that curates it.

## The Viewing Receipt

When you spend money, you get a receipt. When you spend attention, you get nothing. The viewing receipt changes that.

The idea is simple: the station should send you an **email receipt** that accounts for your viewing activity the way a bank statement accounts for your spending. Not to guilt or gamify — but to make consumption visible, reflective, and owned.

A viewing receipt could arrive daily, weekly, or after each session. It might include:

- **What you watched** — The videos you opened, the channels you visited, the time you spent. A clear, honest record of where your attention went.
- **What arrived new** — A summary of fresh content from your subscriptions since the last receipt. The cron layer already tracks this; the receipt simply surfaces it in your inbox.
- **What the curator flagged** — Highlights, recommendations, or insights from the intelligent curator. "Three of your science channels posted this week." "A creator you follow just started a new series." "You have not watched anything from your music subscriptions in two weeks."
- **Viewing patterns** — Over time, receipts accumulate into a personal viewing history. Weekly totals, favourite channels this month, genres that dominate, creators you are drifting away from. Not analytics designed to maximise engagement — analytics designed to help you understand your own habits.
- **What you skipped** — The receipt could note what was available but unwatched, giving a gentle sense of what the station offered versus what was actually consumed.

The receipt should feel like a **letter from your station**, not a notification from an app. It should be:

- **Calm and readable** — Plain text or simple HTML. No tracking pixels, no click-bait subject lines, no dark patterns.
- **Configurable** — Choose daily, weekly, or on-demand. Choose what sections to include. Choose whether to receive it at all.
- **Generated from repo data** — The receipt is assembled by a GitHub Actions workflow that reads the viewing log and subscription state from the repository, formats it, and sends it via a configured email integration (SendGrid, SMTP, or GitHub's own notification system).
- **Archived in the repository** — Each receipt is also committed as a Markdown file, creating a permanent, searchable history of viewing summaries. The inbox copy is ephemeral; the repo copy is forever.

The deeper point is philosophical. Platforms track everything you watch but never give that data back in a form that serves you. YouTube knows your viewing history better than you do, but it uses that knowledge to optimise its own goals. The viewing receipt reclaims that knowledge. It says: this is what I chose to watch, on my terms, summarised in my format, delivered to my inbox, and archived in my repository.

Attention deserves a paper trail. The viewing receipt provides one.

## The YouTube API Surface

The YouTube Data API v3 is the raw material supply chain for the station. Everything the station knows, it knows because the API provided it. Understanding the full API surface is essential to understanding what the station can become.

The following table maps every relevant API resource to its role in the station. This is not a technical specification — it is a statement of ambition. Each row represents a capability the station should be aware of, even if it does not use every one immediately.

| API Resource | What it provides | Station role | Quota cost |
|---|---|---|---|
| **Subscriptions** | List of channels the user subscribes to | The channel lineup — the foundation of the entire station | 1 unit/call |
| **Channels** | Channel metadata: name, description, subscriber count, video count, total views, topic categories, featured channels, branding, upload playlist ID, country, custom URL | Channel identity, tiering data, wall tile content, curator knowledge, network graph of featured channels | 1 unit/call |
| **PlaylistItems** | Videos within a playlist, in order | Playlist blocks, programme sequences, curated collections | 1 unit/call |
| **Playlists** | Playlist metadata: title, description, item count, thumbnails | Playlist identity, programme block headers, collection browsing | 1 unit/call |
| **Videos** | Full video resource: title, description, duration, definition (HD/SD), tags, category, topic details, thumbnails, caption availability, embeddability, content ratings, region restrictions, language, recording location, live streaming status, publish date | The core content record — every cell in the TV guide, every tile on the wall, every item in a receipt is a video resource | 1 unit/call |
| **Videos (statistics)** | View count, like count, comment count | Engagement signals for curation, popularity context, trending detection | 1 unit/call (part of videos.list) |
| **Videos (liveStreamingDetails)** | Actual/scheduled start and end times, concurrent viewers, live chat ID | Live-now detection, premiere countdowns, live badges on the wall, TV guide scheduling | 1 unit/call (part of videos.list) |
| **Videos (player)** | Embed HTML for playback | Inline video playback on the station, channel wall embedded players | 1 unit/call (part of videos.list) |
| **Videos (topicDetails)** | Wikipedia-linked topic categories | Automated genre and topic classification, theme/mood slicing, curator foundation | 1 unit/call (part of videos.list) |
| **Activities** | Channel activity timeline: uploads, likes, favourites, comments, subscriptions, playlist additions | Activity feeds, channel behaviour monitoring, curator awareness of what channels are doing | 1 unit/call |
| **Search** | Find videos, channels, playlists by keyword, topic, location, date, duration, definition, caption availability, event type, language, region | Discovery beyond subscriptions, curator search delegation, TV guide population, content exploration | **100 units/call** |
| **CommentThreads** | Top-level comment threads for a video or channel, with reply previews | Community intelligence, comment highlights, sentiment signals, curator awareness | 1 unit/call |
| **Comments** | Individual comments and replies within a thread | Deeper discussion context, creator responses, community sentiment | 1 unit/call |
| **Captions** | List of caption tracks for a video (language, type, auto-generated or manual) | Transcript availability detection, language metadata, accessibility information | 50 units/call |
| **Captions (download)** | Actual caption/transcript text in SRT, VTT, or other formats | Searchable transcript archives, AI summarisation, content analysis, the station's deepest knowledge of what was said | Requires OAuth; 200 units/call |
| **VideoCategories** | YouTube's official video category taxonomy (per region) | Baseline categorisation, genre row headers, category-based filtering | 1 unit/call |
| **ChannelSections** | How a channel organises its own page: featured channels, playlists, uploads | Understanding how creators see their own channel, discovering related channels | 1 unit/call |
| **I18nLanguages** | List of languages YouTube supports | Language filtering, interface localisation, multilingual content awareness | 1 unit/call |
| **I18nRegions** | List of regions/countries YouTube supports | Regional content awareness, region-specific category availability | 1 unit/call |

### Quota as editorial budget

The default daily quota is 10,000 units. This is not generous. A single search costs 100 units. Listing captions costs 50. But listing videos, channels, playlists, activities, and comments costs just 1 unit each.

This means the station's cron budget naturally favours deep knowledge of subscribed content (cheap) over broad discovery searches (expensive). That is a feature, not a limitation. The station is about owned curation first and open discovery second. The quota structure aligns with that philosophy.

A well-designed cron pipeline could, within 10,000 daily units:
- Refresh all subscriptions (1 call)
- Update channel metadata for 100 channels (100 calls = 100 units)
- Fetch latest videos for 100 channels via upload playlists (100 calls = 100 units)
- Retrieve full video details for 500 new videos (5 calls at 50 videos each = 5 units)
- Poll activity feeds for 50 priority channels (50 calls = 50 units)
- Fetch comment threads for 20 notable videos (20 calls = 20 units)
- Check caption availability for 50 videos (1 call at 50 units)
- Run 10 targeted searches for the curator (10 calls = 1,000 units)
- Leave thousands of units in reserve for live checks, statistics updates, and on-demand queries

The point is not to optimise every unit. The point is to understand that the quota is a budget, and the station should spend it deliberately — the same way it spends attention.

### Authentication levels

Not all API capabilities are available with a simple API key:

| Access level | What it unlocks | Station implication |
|---|---|---|
| **API key only** | Public video/channel/playlist data, search, categories, reference data | The foundation — enough for the core station experience |
| **OAuth 2.0 (read)** | User's own subscriptions, liked videos, private playlists, watch history (limited), activities | Personalisation — the station becomes truly personal, drawing from the user's own account |
| **OAuth 2.0 (write)** | Caption downloads for own videos, playlist management, subscription management | Deep features — transcript access for own content, managing playlists from the station |

The station should work with just an API key for public data, but become richer with OAuth when the viewer chooses to authenticate. Authentication is opt-in depth, not a gate.

## The Data Model Aspiration

The most important asset this project can produce is not the website. It is the dataset.

The repo should gradually become a high-quality personal record of:

- Which channels I subscribe to — and their full metadata: descriptions, subscriber counts, video counts, topic categories, featured channels, publishing cadence, and branding
- Which videos those channels publish — with the full resource: title, description, duration, definition, tags, category, topic details, thumbnail URLs, caption availability, embeddability, region restrictions, language, and recording location where available
- How those videos are received — view counts, like counts, comment counts: engagement data captured over time so trends are visible, not just snapshots
- Which playlists I maintain or care about — with their items, ordering, and metadata
- Which videos I have liked
- What subscribed channels have been doing — activity feeds tracking uploads, likes, favourites, playlist additions, and new subscriptions as a timeline of channel behaviour
- What people are saying — archived samples of comment threads for significant videos, capturing community discussion as structured data
- What was said in the videos themselves — caption transcripts archived as searchable text, turning hours of video into a queryable library of spoken content
- Which categories and topics apply — both YouTube's own video categories and topic detail links, and the richer AI-generated tags from the curator
- What is happening live — which channels are currently broadcasting, which streams are scheduled, and a historical record of live events
- Which items are new, recurring, featured, archived, or manually highlighted
- Which videos were actually watched, when, and for how long — the viewing log that powers receipts and reflection
- Which viewing receipts were sent, and what patterns they revealed over time
- What the TV guide looked like — archived programme guide snapshots that capture what was available on a given day, creating a historical record of the station's schedule

Good structured data makes everything else possible. If the underlying resources are clean, stable, and versioned, then the interface can evolve freely without losing the real value.

## The Editorial Standard

The station should feel authored.

That means the project should prefer:

- **Intentional curation over maximal ingestion**
- **Readable pages over clever interfaces**
- **Stable structures over brittle automation**
- **Human legibility over hidden magic**

The repository should not become a generic mirror of everything YouTube exposes. It should become a selective, expressive representation of the parts that matter.

## The Philosophical Point

This is not just a tool. It is a statement about how content consumption could work differently.

Today, platforms give us feeds but own the experience. They choose the layout, the ranking, the defaults, the surrounding noise, and the behavioral incentives. Even when the underlying content is excellent, the interface is designed for the platform's goals first.

This project argues for a different model:

- Let the platform host the content.
- Let the user own the organisation.
- Let the pipeline be inspectable.
- Let the interface be reshapeable.
- Let taste become infrastructure.

Even if this only serves one person, it is still meaningful. A personal system can be a real alternative, not just a prototype.

## The Standard for Future Work

Before building any feature, it should be possible to ask:

1. Does this make the station more personal?
2. Does this increase ownership or clarity?
3. Does this improve the quality of the data layer?
4. Does this keep the boundary between source, processing, and presentation clean?
5. Does this make the experience calmer, clearer, or more intentional?
6. Does this work when the station is displayed across multiple screens?
7. Does this respect the cron processing budget and API quota costs?
8. Does this make the intelligent curator smarter or more helpful?
9. Does this contribute to a meaningful viewing receipt or help the viewer reflect on their habits?
10. Does this use the YouTube API surface wisely — drawing on the right resource (video details, statistics, captions, comments, activities, search, live data, categories) for the right purpose, without fetching data that will never be used?
11. Does this make the TV guide more complete, more accurate, or more useful as a programme schedule?

If the answer is no, the feature is probably a distraction.
