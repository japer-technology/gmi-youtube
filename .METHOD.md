# Method

## Primary Reference

Read `.ASPIRATION.md` first.

That document is the primary source for what this repository is trying to become. This document is secondary and exists to describe how implementation should proceed in service of that aspiration.

If there is any tension between the two documents, `.ASPIRATION.md` wins.

## Why This Exists

`.ASPIRATION.md` explains what this repository should become.

This document explains how an AI Coding Agent should build it without losing the qualities that make the aspiration worth pursuing: ownership, clarity, calmness, inspectability, and durable value.

The method is simple:

1. establish the right boundaries first
2. build the data layer before the interface depends on it
3. prefer static outputs over hidden runtime state
4. ship in narrow, legible increments
5. keep every step auditable in-repo

If the aspiration is the product vision, the method is the construction order.

## Delivery Rules

The AI Coding Agent should follow these rules throughout implementation:

- **Repository first** — derived resources in the repository are the source of truth for the product
- **Static first** — prefer GitHub Pages-compatible outputs over server-heavy designs
- **Schema first** — define stable resource shapes before building views that consume them
- **Workflow first** — prove ingestion and publishing paths early so the system becomes operable, not merely imagined
- **Editorial first** — optimise for useful curation and legibility, not maximal feature volume
- **Incremental first** — each change should leave the repository more usable than before

## Recommended Build Order

Implementation should proceed in this sequence:

1. **Scaffold the repository for the station**
2. **Define stable schemas and example resources**
3. **Implement one narrow ingestion flow from YouTube into repository resources**
4. **Publish one static interface that reads those resources**
5. **Add scheduling, history, and receipts**
6. **Expand toward guide, wall, search, and curator capabilities**

Only the first phase is specified in detail here. It is the opening move, and it should be completed before deeper feature work begins.

## Opening Scaffolding Steps

The opening scaffold should create a software shape that matches the aspiration before any large-scale ingestion or interface work starts.

### Step 1: Establish the Runtime and Project Conventions

The repository already includes Bun-based tooling under `.github-minimum-intelligence/`. Reuse that direction rather than introducing a second runtime strategy.

The AI Coding Agent should:

1. create a root `package.json` for the station work if one does not already exist
2. use **Bun + TypeScript** for scripts and build logic
3. keep scripts explicit and small
4. avoid introducing a database, backend service, or opaque deployment platform

Minimum expected scripts:

- `check`
- `build`
- `ingest`
- `publish`

These scripts do not need to be fully powerful on day one, but the command surface should exist early so later work has a stable operating shape.

### Step 2: Create the First-Class Repository Boundaries

The initial scaffold should make the system legible by giving each responsibility an obvious home.

Create top-level directories for:

| Path | Purpose |
|---|---|
| `schemas/` | versioned resource definitions for stable station data |
| `resources/` | derived JSON and Markdown artifacts committed to the repository |
| `scripts/` | Bun/TypeScript ingestion, transform, validation, and publishing scripts |
| `site/` | static GitHub Pages application |
| `tests/fixtures/` | small, deterministic example inputs and outputs for pipeline work |
| `.github/workflows/` | scheduled and on-demand automation entry points |

The naming may evolve later, but the boundary discipline should not.

### Step 3: Define the Minimum Viable Data Contracts

Before fetching large amounts of YouTube data, define the smallest useful station resources.

The first scaffold should include schemas for at least:

- `channel`
- `video`
- `playlist`
- `guide-entry`
- `viewing-receipt`

Each schema should state:

- identifier strategy
- source provenance
- update timestamp
- fields required for static rendering
- fields required for later editorial enrichment

The first example resources should be hand-authored fixtures if necessary. Real ingestion can come immediately after, but the interface and workflows should not start from undefined shapes.

### Step 4: Create a Thin Static Site Shell

Do not wait for full data richness before making the product visible.

The first scaffold for `site/` should provide:

- a home page that explains the station
- a placeholder guide view
- a placeholder channel wall view
- a simple resource-loading path from `resources/`

This shell should prove that:

1. GitHub Pages can render the station
2. static resources can be consumed without runtime ambiguity
3. later guide and wall work has a concrete surface to improve

The opening shell should be visually calm and structurally simple.

### Step 5: Add the Initial Workflow Spine

The first workflows should be modest and explicit.

Create workflow entry points for:

- manual validation
- manual ingest
- scheduled ingest
- static site publish

At the beginning, manual dispatch is more important than ambitious scheduling. The system should first prove that it can run safely, deterministically, and transparently.

Each workflow should make these things visible:

- what input it used
- what resources it changed
- what artifact or site output it produced
- whether quota-sensitive operations were executed

### Step 6: Seed the Repository With Example State

A station is easier to build when there is already something to inspect.

The first scaffold should include:

- one example channel record
- one example video record
- one example playlist record
- one example guide dataset
- one example viewing receipt

These examples should be realistic enough to drive interface and validation work, but small enough to understand instantly in code review.

### Step 7: Define the First Validation Loop

The opening scaffold is incomplete until the repository can check itself.

The AI Coding Agent should add validation that confirms:

- schemas are valid
- example resources conform to schemas
- the site can build from repository resources
- workflows invoke documented commands rather than hidden ad hoc logic

Validation should remain simple. It only needs to prove that the station has a trustworthy spine.

### Step 8: Document Secrets, Quota, and Safety Boundaries

The aspiration depends on authenticated YouTube access, scheduled automation, and durable repository state. Those capabilities should be documented before they are used heavily.

The scaffold should document:

- required YouTube credentials and where they live
- which tasks require OAuth versus API key access
- the expected daily quota budget
- which commands are safe to run locally
- which workflows are allowed to write repository resources

If an operation changes repository state, publishes the site, or spends meaningful API quota, that should be obvious from the documentation and workflow names.

## Definition of Done for the Opening Scaffold

The opening scaffolding phase is complete when all of the following are true:

- the repository has clear directories for schemas, resources, scripts, and site code
- the station has a root runtime and command surface
- at least a few stable schemas and example resources exist
- a static site shell can read repository resources
- workflows can validate, ingest in a narrow way, and publish
- the system's secrets, quota assumptions, and write paths are documented

At that point, the repository is no longer just an idea with a vision document. It is a buildable station foundation.

## What Should Happen Next

After the opening scaffold is complete, the next method section should document the first real ingestion slice:

1. pull subscriptions or one curated playlist from YouTube
2. normalize the response into repository resources
3. publish that state into a usable guide or wall view
4. archive the run with clear provenance

That is the first moment where aspiration becomes operation.
