## Why GitHub Pages Is the Tour de Force

The aspiration describes four layers: YouTube as the content source, GitHub as the processing and memory layer, GitHub Pages as the experience layer, and the AI Agent as the conversational interface.

Of these four, GitHub Pages is the layer the user actually sees. Every other layer exists in service of what appears on that screen. The ingestion pipeline, the schemas, the curator logic, the multi-screen coordination — all of it converges on the static site that GitHub Pages delivers.

If the GitHub Pages experience is ordinary, the entire system feels like an engineering exercise. If the GitHub Pages experience is excellent — calm, fast, legible, personal, and genuinely better than opening YouTube directly — then the system delivers on the aspiration.

The last steps exist to close that gap.

## Phase 9 — Station Identity

The station currently functions but lacks visual identity. Every page says "Station" but nothing about the experience communicates that this is a personal, intentionally crafted viewing environment. Identity is what distinguishes a working prototype from a product someone returns to daily.

### Step 9.1: Implement Dark Mode

The aspiration describes editorial blocks including "Late" (22:00–02:00) characterised as "calm, ambient, rewatchable." A personal TV station used in the evening must support dark mode — not as a novelty toggle, but as a core viewing context.

Implementation:

- define a `prefers-color-scheme: dark` media query with a complete dark palette
- add a manual theme toggle in the site header that persists via `localStorage`
- ensure every page — guide, wall, receipt, search, curator — renders correctly in both modes
- use CSS custom properties already established in `style.css` (`--bg`, `--fg`, `--muted`, `--border`, `--accent`, `--card-bg`) so the switch is comprehensive
- dark mode should feel calm and low-contrast, not merely inverted — appropriate for a late-night viewing environment

### Step 9.2: Add Favicon and Web App Manifest

A station that can be bookmarked, pinned, or added to a home screen needs an icon and a manifest.

Implementation:

- create a simple, recognisable favicon (SVG preferred for scalability, with PNG fallback)
- add a `site.webmanifest` to `site/` with station name, short name, theme colour, background colour, display mode (`standalone`), and icon references
- add `<link rel="icon">` and `<link rel="manifest">` to every HTML page
- set `<meta name="theme-color">` to match the station's accent colour, with a dark variant for dark mode

### Step 9.3: Add Open Graph and Social Meta Tags

When a station link is shared, it should present clearly — not as a generic "Station" page.

Implementation:

- add `<meta property="og:title">`, `og:description`, `og:type`, and `og:image` to each page
- add `<meta name="description">` for search engine context
- use distinct titles per page: "TV Guide — Station", "Channel Wall — Station", etc. (some pages already have this; make it consistent)
- the OG image can reference the repository's existing `GitHub-YouTube.png` or a purpose-built station card

### Step 9.4: Add an About Page

The `index.html` home page includes a brief "What Is This?" section, but a visitor or future-self deserves a fuller explanation of the system's architecture, principles, and provenance — without requiring them to read `.ASPIRATION.md`.

Implementation:

- create `site/about.html` with a clear, non-technical explanation of the station
- include: what the station does, where the data comes from, how the pipeline works at a high level, and why the project exists
- link to it from the main navigation
- include a link to the repository for full transparency
- add `data-screen-role="about"` to the body element for consistency with the multi-screen architecture

## Phase 10 — Experience Polish

The functional foundation is solid. Polish is what makes the difference between a system that works and one that a person wants to use.

### Step 10.1: Improve Loading States

Every page currently shows `Loading guide data…` or similar text placeholders while resources load. These should feel intentional rather than broken.

Implementation:

- replace text placeholders with subtle skeleton loaders — light pulsing rectangles that hint at the content shape
- add CSS keyframe animations for the skeleton pulse (no JavaScript dependency)
- ensure skeletons match the layout of the content they replace: card shapes for the guide, grid shapes for the wall, list shapes for receipts
- keep skeletons visually minimal — this is a calm station, not a loading spectacle

### Step 10.2: Add Keyboard Navigation

A personal TV station should be operable without reaching for a mouse — especially in a multi-screen context where one screen is across the room.

Implementation:

- `/` or `Ctrl+K` focuses the search input on the search page (or globally navigates to search)
- `←` and `→` navigate guide dates on the guide page
- `Escape` dismisses the playback notification bar, closes receipt detail view, or clears search
- `?` shows a brief keyboard shortcut overlay
- all shortcuts should be non-intrusive — they enhance, but nothing breaks without them
- shortcuts should be documented in the about page or help overlay

### Step 10.3: Add Freshness Indicators

A personal station should make it immediately obvious whether its data is current. When did the last ingestion run? How old is the guide data? Is the station operating or stale?

Implementation:

- display a "Last updated" timestamp in the site footer, derived from the build timestamp or the most recent resource modification date
- on the guide page, indicate whether the guide is for today, or if the most recent guide is from a past date
- on the wall page, show a subtle freshness badge when a channel's most recent video is more than a week old
- these indicators should be informative, not alarming — calm awareness, not urgency

### Step 10.4: Improve Accessibility

The station should be usable by everyone, including those using screen readers or keyboard-only navigation.

Implementation:

- add ARIA landmark roles to header, main, and footer where not already present
- ensure all interactive elements (buttons, links, toggles) have clear accessible names
- add `aria-live="polite"` to dynamic content regions (guide grid, search results, receipt detail) so screen readers announce updates
- ensure colour contrast meets WCAG AA in both light and dark modes
- add skip-to-content links for keyboard users
- test focus order on each page — tab order should follow visual reading order

### Step 10.5: Print-Friendly Receipts

Viewing receipts answer: "What did I watch this week?" That answer should be printable — a calm summary suitable for reflection away from a screen.

Implementation:

- add a `@media print` stylesheet that strips navigation, adjusts layout to single-column, and presents the receipt detail cleanly
- hide the receipt list when a detail view is active in print
- ensure receipt stats, channel distribution, and video lists render clearly on paper
- include the station name and period in the print header

## Phase 11 — Progressive Web App

The aspiration calls for a system that "works well across devices and screens." A Progressive Web App turns a static site into something that can be installed, cached, and used with near-native reliability.

### Step 11.1: Add a Service Worker

A service worker enables offline access to previously loaded station pages and resources.

Implementation:

- create a minimal `sw.js` in `site/` that caches the HTML shell, CSS, JavaScript, and the most recent resource manifest
- use a cache-first strategy for static assets (HTML, CSS, JS) and a network-first strategy for JSON resources
- when offline, serve cached resources and display a subtle "Offline — showing cached data" indicator
- the service worker should be registered from `station.js` on first load
- keep the service worker simple — this is a static site, not a complex application

### Step 11.2: Support Installability

With the web app manifest from Step 9.2 and a service worker, the station becomes installable on mobile and desktop.

Validation:

- confirm the manifest includes required fields: `name`, `short_name`, `start_url`, `display`, `icons`, `theme_color`, `background_color`
- confirm the service worker is registered and functional
- test the install prompt on Chrome, Edge, and Safari
- the installed station should open in standalone mode without browser chrome

## Phase 12 — Documentation and Showcase

The repository README is minimal. The station's GitHub Pages site is its public face, but the repository itself should explain what the station is and how to use it — both for the owner and for anyone who encounters it.

### Step 12.1: Update the README

The current README points to `.github-minimum-intelligence/README.md` and contains a logo. It should introduce the station.

The updated README should include:

- a one-paragraph description of what the station is
- a link to the live GitHub Pages site
- a brief architecture summary (YouTube → GitHub Actions → Repository Resources → GitHub Pages)
- a table of available commands (`check`, `build`, `ingest`, `generate-guide`, `generate-receipt`, `curator`, `publish`)
- a link to `OPERATIONS.md` for credential and quota details
- a link to the design documents (`.ASPIRATION.md`, `.METHOD.md`, `.NEXT-STEPS.md`, `.LAST-STEPS.md`)
- the existing link to `.github-minimum-intelligence/README.md`

### Step 12.2: Add a Station Health Dashboard

The station operates autonomously through scheduled GitHub Actions. It should be easy to verify that the system is healthy.

Implementation:

- add a lightweight `status.html` page (or a section on the home page) that displays:
  - total channels tracked
  - total videos in the library
  - date of the most recent guide
  - date of the most recent receipt
  - date of the most recent ingestion (from build metadata or commit timestamps)
- this dashboard reads from the existing resource manifest — no new infrastructure required
- add `data-screen-role="status"` for multi-screen consistency

### Step 12.3: Add Screenshot Documentation

The station should document its own visual appearance so that the design intent is preserved even as the interface evolves.

Implementation:

- capture representative screenshots of each page: home, guide, wall, search, receipt, curator
- include both light and dark mode variants
- store screenshots in a `docs/screenshots/` directory
- reference them from the README

## Guiding Principles for the Last Steps

These principles carry forward from `.ASPIRATION.md`, `.METHOD.md`, and `.NEXT-STEPS.md`:

1. **The experience layer is the product.** Every infrastructure phase exists in service of what appears on the GitHub Pages screen. These last steps honour that.
2. **Calm over clever.** Dark mode should be restful. Loading states should be subtle. Keyboard shortcuts should be discoverable but unobtrusive. Nothing should compete with the content.
3. **The station should feel finished.** A favicon, a manifest, an about page, a dark mode — these are not features. They are the difference between a project and a product.
4. **Accessibility is not optional.** A personal TV station should work for the person who built it, however they interact with screens.
5. **Static remains the strategy.** The service worker caches a static site. The health dashboard reads static JSON. No new runtime complexity is introduced.
6. **GitHub Pages is the tour de force.** When someone opens the station URL, they should see a calm, personal, intelligently curated viewing environment that makes YouTube better — not a developer prototype that happens to show some data.

## Definition of Done for the Last Steps

The last steps are complete when:

- the station has a clear visual identity with dark mode, favicon, and branding
- every page loads gracefully with skeleton states instead of text placeholders
- keyboard navigation works across all pages
- the site is installable as a Progressive Web App with offline access
- accessibility meets WCAG AA standards in both light and dark modes
- receipts print cleanly
- the README introduces the station and links to the live site
- a health dashboard shows the station's operational status
- the experience is calm, fast, personal, and genuinely preferable to opening YouTube directly for many viewing tasks

That is the point where the tour de force is delivered.
