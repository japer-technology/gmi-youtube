## Phase 1 — First Real Ingestion Slice

This is the phase `.METHOD.md` identifies explicitly as the first post-scaffold work. It is the moment where aspiration becomes operation.

### Step 1.1: Ingest a Real Channel

Run the existing `channel:<id>` ingestion scope against a real YouTube channel using a live API key.

Confirm that:

- `resources/channels/` receives a new JSON file matching the channel schema
- `resources/videos/` receives JSON files for recent uploads matching the video schema
- quota usage is logged accurately
- the workflow commits results with clear provenance

This step validates that the ingestion pipeline works end-to-end with real data.

### Step 1.2: Ingest a Real Playlist

Run the existing `playlist:<id>` ingestion scope against a real YouTube playlist.

Confirm that:

- `resources/playlists/` receives a new JSON file matching the playlist schema
- `resources/videos/` receives JSON files for all playlist videos
- batch handling works correctly for playlists with more than 50 items
- quota usage stays within budget for the operation

### Step 1.3: Generate a Guide From Ingested Data

Build a guide generation step that takes ingested video resources and produces a guide-entry array for a given date.

The generator should:

1. read all video resources from `resources/videos/`
2. sort by `publishedAt` descending
3. assign `scheduledAt` timestamps across the day in reasonable editorial blocks
4. write the result to `resources/guide/<date>.json`

The guide does not need to be editorially brilliant yet. It needs to be structurally correct and populated from real data.

### Step 1.4: Publish the Ingested State to the Site

Run `bun run build` and confirm that:

- `dist/` contains the site with real resource data
- `dist/resources/` contains the manifest and all ingested resources
- the guide page renders real entries
- the index page shows real tracked channels

### Step 1.5: Archive the First Real Run

Confirm that the ingest workflow:

- commits changed resources to the repository
- the commit message identifies the scope, date, and quota spent
- the git history shows a clean, inspectable record of what changed

This completes the first real ingestion cycle. The station now has real data flowing through the entire pipeline.

## Phase 2 — Subscription Intelligence

Subscriptions are the highest-value signal for a personal station. They represent years of deliberate curation by the user and should be the primary source for channel tracking.

### Step 2.1: Implement OAuth Credential Flow

Add OAuth 2.0 support for authenticated YouTube API access.

Requirements:

- store `YOUTUBE_CLIENT_ID`, `YOUTUBE_CLIENT_SECRET`, and `YOUTUBE_REFRESH_TOKEN` as repository secrets
- implement token refresh logic in the ingestion script
- document the one-time OAuth consent process in OPERATIONS.md
- ensure the system falls back gracefully to API-key-only mode when OAuth is unavailable

### Step 2.2: Ingest Subscriptions

Implement the `subscriptions` scope in `scripts/ingest.ts`.

The ingestion should:

1. call `subscriptions.list` with the authenticated user context
2. page through the full subscription list
3. normalize each subscription into a channel resource
4. write or update `resources/channels/<id>.json` for each subscribed channel
5. log quota usage (1 unit per page of results)

### Step 2.3: Build the Subscription Graph

After subscriptions are ingested, build a subscription index:

- create `resources/subscriptions.json` listing all subscribed channel IDs with metadata
- add a `subscriptions` schema to `schemas/` defining the shape
- use this index as the primary input for scheduled ingestion of recent uploads

### Step 2.4: Schedule Subscription-Driven Ingestion

Extend `scheduled-ingest.yml` to:

1. read the subscription index
2. fetch recent uploads for each subscribed channel (using the channel's uploads playlist)
3. normalize new videos into `resources/videos/`
4. generate a fresh guide for the current date
5. commit and publish

This is the point where the station begins to operate autonomously. New videos from subscribed channels appear in the guide without manual intervention.

## Phase 3 — TV Guide Intelligence

The guide should evolve from a chronological video list into an editorially structured programme schedule.

### Step 3.1: Implement Editorial Blocks

Define named editorial blocks with time boundaries:

| Block | Time Window | Character |
|---|---|---|
| Morning | 06:00–10:00 | Short-form, news, briefings |
| Midday | 10:00–14:00 | Medium-form, tutorials, talks |
| Afternoon | 14:00–18:00 | Long-form, documentaries, deep dives |
| Evening | 18:00–22:00 | Flagship content, curated highlights |
| Late | 22:00–02:00 | Calm, ambient, rewatchable |

The guide generator should assign videos to blocks based on duration, topic, and recency. The block names and time boundaries should be configurable in a `resources/guide-config.json` file.

### Step 3.2: Render the Guide as a Time Grid

Enhance `site/guide.html` and `station.js` to render the guide as a channel-by-time grid:

- rows represent channels or editorial blocks
- columns represent time slots
- cells link to video playback
- live and upcoming content is visually distinguished
- the current time is highlighted

### Step 3.3: Add Live and Premiere Detection

Extend the ingestion pipeline to check `liveStreamingDetails` for each video:

- flag videos that are currently live
- flag videos with scheduled premiere times
- surface these prominently in the guide with clear visual indicators

### Step 3.4: Support Guide History

Archive each day's guide as a permanent resource:

- `resources/guide/<date>.json` is never overwritten after the day passes
- historical guides can be browsed in the site
- this builds the temporal memory that makes receipts and reflection possible

## Phase 4 — Channel Wall

The channel wall should become a configurable, responsive matrix for rapid situational awareness.

### Step 4.1: Implement Configurable Wall Layouts

Add a `resources/wall-layouts.json` file that stores named layout configurations:

```
[
  { "name": "default", "rows": 3, "cols": 3, "channels": [...] },
  { "name": "science", "rows": 2, "cols": 3, "channels": [...] },
  { "name": "calm", "rows": 1, "cols": 1, "channels": [...] }
]
```

Add a `wall-layout` schema to `schemas/` defining the shape.

### Step 4.2: Render Responsive Wall Grids

Enhance `site/wall.html` and `station.js` to:

- read the selected layout from `wall-layouts.json`
- render an N×M grid of channel tiles
- show the most recent or currently live video for each channel
- include clear live, upcoming, and stale-state indicators
- support layout switching through the interface

### Step 4.3: Support Embedded Playback

Where YouTube embed permissions allow:

- embed the player directly in wall tiles
- fall back to thumbnail + link when embedding is restricted
- respect user preferences for autoplay behaviour

## Phase 5 — Viewing Receipts

Receipts close the feedback loop. They turn viewing into a reflective practice rather than a passive consumption metric.

### Step 5.1: Build the Receipt Generation Pipeline

Create a `generate-receipt` script that:

1. reads the guide for a given period (day or week)
2. compares guide entries against any available watch signals
3. identifies what arrived (new videos from subscriptions), what was featured (guide entries), and what was skipped
4. writes a receipt to `resources/receipts/receipt-<date>.json`

Initially, watch signals may not be available from the API. The receipt can still report what arrived and what was curated.

### Step 5.2: Add Receipt Rendering

Add a `site/receipt.html` page that:

- renders a receipt for a selected period
- shows watched, arrived, skipped, and curator-flagged content
- displays stats (total duration, channel distribution, topic spread)
- includes curator notes when available

### Step 5.3: Schedule Daily and Weekly Receipts

Add workflow steps that:

- generate a daily receipt at the end of each day
- generate a weekly receipt summarising the past seven days
- commit receipts to the repository for permanent archival

## Phase 6 — Transcript and Search Intelligence

Transcripts are the bridge between video as media and video as searchable knowledge.

### Step 6.1: Ingest Caption Availability

Extend video ingestion to call `captions.list` for each video:

- record available caption tracks and languages in the video resource
- flag videos where captions are available for download
- respect quota discipline — captions.list costs 50 units per call, so apply selectively

### Step 6.2: Download and Store Transcripts

For high-priority videos (curator-flagged, long-form, or explicitly requested):

- download captions using `captions.download` (requires OAuth, 200 units per call)
- store transcripts as `resources/transcripts/<videoId>.txt`
- add a `transcript` schema to `schemas/`
- track transcript availability in the video resource

### Step 6.3: Build a Search Index

Create a lightweight search index from available transcripts and video metadata:

- index video titles, descriptions, tags, and transcript text
- store the index as a static JSON file in `dist/`
- support keyword search from the site interface

### Step 6.4: Add Search to the Site

Add a `site/search.html` page that:

- accepts a text query
- searches the static index
- returns ranked results with context snippets
- links directly to playback with relevant timestamps when transcript data is available

## Phase 7 — Intelligent Curator

The curator is the conversational layer that transforms a structured data system into a personally operated media environment.

### Step 7.1: Define Curator Actions

Document the specific actions the AI Agent should be able to perform:

| Action | Input | Output |
|---|---|---|
| Build a guide | Date, preferences | `resources/guide/<date>.json` |
| Update a wall layout | Layout name, channel list | `resources/wall-layouts.json` |
| Recommend content | Topic, mood, duration | Ranked video list |
| Summarise a channel | Channel ID | Narrative summary with recent activity |
| Generate a receipt | Period | `resources/receipts/receipt-<date>.json` |
| Flag content | Video ID, reason | Tag added to video resource |
| Answer a question | Natural language query | Grounded response from repository state |

### Step 7.2: Implement Curator Data Access

Give the AI Agent structured access to repository state:

- resource reading (channels, videos, playlists, guides, receipts)
- resource writing (guide entries, wall layouts, video tags, receipts)
- schema validation before any write
- quota-aware YouTube API access for fresh data

### Step 7.3: Implement Curator Logic for Guide Generation

The curator should be able to:

1. assess available content across subscribed channels
2. classify videos by topic, mood, format, and estimated quality
3. propose an editorial schedule that balances variety, recency, and user preferences
4. write the schedule as a guide resource

### Step 7.4: Implement Curator Logic for Content Discovery

The curator should be able to:

1. accept natural language queries ("Find a recent deep interview on AI safety")
2. search repository resources and, when necessary, the YouTube API
3. return grounded results with provenance
4. distinguish between repository-known content and fresh API results

### Step 7.5: Implement Curator Feedback Loop

The curator should learn from receipts and viewing patterns:

- note which curated content was watched versus skipped
- adjust future recommendations based on demonstrated preferences
- surface observations in curator notes on receipts

## Phase 8 — Multi-Screen Coordination

Multi-screen support should improve the experience without introducing distributed-systems complexity.

### Step 8.1: Implement Screen Roles

Define distinct screen roles:

- **Guide screen** — displays the programme guide
- **Playback screen** — plays the selected video
- **Wall screen** — displays the channel wall
- **Curator screen** — hosts the conversational curator interface

### Step 8.2: Add Lightweight Coordination

Implement coordination using browser-native mechanisms:

- `BroadcastChannel` for same-origin tab communication
- URL state for shareable screen configurations
- `localStorage` for persisting screen role preferences

When a user selects a video on the guide screen, the playback screen should respond. This coordination should be an enhancement, not a requirement — each screen should work independently.

### Step 8.3: Responsive Layout Adaptation

Each screen role should adapt to its display context:

- guide on a narrow screen collapses to a list
- wall on a wide screen expands to a full grid
- playback on a mobile screen fills the viewport
- curator on any screen presents a clean conversation interface

## Guiding Principles for Execution

These principles carry forward from `.ASPIRATION.md` and `.METHOD.md`:

1. **Ship in narrow, legible increments.** Each step should leave the repository more usable than before.
2. **Prove operation before expanding scope.** A working ingestion cycle matters more than a complete feature set.
3. **Treat quota as an editorial budget.** Every API call should have a clear purpose and documented cost.
4. **Keep the data layer durable.** Interfaces can be redesigned. The dataset and its history are the permanent asset.
5. **Prefer static outputs over hidden state.** If it cannot be inspected in the repository, it should not be trusted.
6. **Make the AI Agent effective, not decorative.** The curator should perform concrete actions grounded in repository state, not produce vague suggestions.
7. **Optimise for calm, deliberate use.** Every feature should reduce noise, not add it.
